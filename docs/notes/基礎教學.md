⏺ 白話解釋：從 C 到 FPGA 晶片                                                        
                                                                                   
  你寫的 C code 到底在幹嘛？                                                         
                                                                                     
  想像你在設計一台5G 手機的基頻晶片。手機收到無線電訊號後，需要做一連串「數學運算」才
  能變成你聽到的聲音或看到的影片。                                                   
                                                                                     
  這些數學運算就是我們寫的那些 C kernel：                                            
   
  手機收到無線電波                                                                   
      ↓                                                           
  FIR filter（過濾雜訊）← 我們的 fir_5g.c
      ↓                                                                              
  FFT（頻域轉換，OFDM 解調）← 我們的 fft.c
      ↓                                                                              
  LDPC decoder（修復傳輸錯誤）← 我們的 ldpc_decoder.c             
      ↓                                                                              
  你看到乾淨的影片                                                
                                                                                     
  為什麼要用 FPGA？                                               
                                                                                     
  這些運算量巨大（5G 每秒處理幾十億次），用一般 CPU                                  
  太慢。所以聯發科/高通把這些運算「燒進」專用晶片（FPGA 或 ASIC）。
                                                                                     
  但從 C code 變成晶片電路，中間需要一個「翻譯」步驟：                               
   
  C code（你寫的演算法）                                                             
      ↓  HLS 工具（Vitis HLS）                                    
      ↓  + #pragma HLS（告訴翻譯器怎麼優化）                                         
  硬體電路（Verilog）                                                                
      ↓  佈線工具（Vivado）                                                          
  FPGA bitstream（可以燒進晶片的檔案）                                               
                                                                                     
  #pragma HLS 是什麼？（用餐廳比喻）                                                 
                                                                                     
  你的 C code 像是一份菜單（要做什麼菜）。#pragma HLS 像是告訴廚房的指令：           
                                                                  
  // 菜單：做 128 碗湯                                                               
  for (i = 0; i < 128; i++) {                                                        
      #pragma HLS PIPELINE II=1    // ← 「用流水線！第一碗還沒裝完，第二碗就開始煮」
                                                                                     
      for (j = 0; j < 16; j++) {                                                     
          #pragma HLS UNROLL factor=8  // ← 「同時派 8 個廚師做這步」                
          acc += input[i-j] * coeffs[j];                                             
      }                                                                              
  }                                                                                  
                                                                                     
  #pragma HLS ARRAY_PARTITION variable=coeffs complete            
  // ↑ 「把調味料分裝到 16 個小碗，每個廚師各拿一碗，不用排隊搶」
                                                                                     
  不加 pragma：一個廚師一碗一碗做 → 2050 個時鐘週期                                  
  加了 pragma：8 個廚師流水線做 → 1028 個時鐘週期（快 2 倍）                         
  加到最激進：16 個廚師全展開 → 64 個時鐘週期（快 32 倍）                            
                                                                                     
  我們的實驗做了什麼？                                                               
                                                                                     
  第一步：寫 C code（FIR, GEMM, DCT, LDPC 等 17 個 kernel）                          
           ↓                                                                         
  第二步：讓 LLM（Gemma4 / Qwen）建議 pragma                                         
           「你覺得這段 code 該怎麼優化？」                                          
           LLM：「用 PIPELINE II=1, UNROLL factor=8, PARTITION coeffs complete」     
           ↓                                                                         
  第三步：post-process merge（安全合併）                                             
           只把 LLM 建議的 pragma 行放回去，其他 code 一字不動                       
           → 保證 100% 不會改壞原始程式                                              
           ↓                                                                         
  第四步：丟到 AWS 上的 Vitis HLS 合成                                               
           Vitis 把「C + pragma」翻譯成硬體電路                                      
           報告：「這個設計需要 1028 個 cycles，用了 1672 個 LUT，6 個 DSP」         
           ↓                                                                         
  第五步：比較 baseline vs LLM 建議                                                  
           「沒有 pragma → 2050 cycles」vs「有 LLM pragma → 1028 cycles」            
           → Speedup = 2.0x                                                          
                                                                                     
  AWS 在這裡扮演什麼角色？
