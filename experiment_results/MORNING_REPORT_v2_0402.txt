## 二、PIE tgt_code 分析（關鍵發現）

分析 200 個 PIE 人類優化案例的 delta：
  - I/O 加速（cin→scanf, endl→\n）: 73%
  - 演算法改變: 26%
  - 資料結構替換: 1%

具體 I/O pattern (200 sample):
  - cin → scanf: 82 題 (41%)
  - cout → printf: 82 題 (41%)
  - endl → \n: 66 題 (33%)
  - string → char[]: 26 題 (13%)
  - 加 sync_with_stdio: 21 題 (11%)

結論：PIE 的 3.56x speedup 主要來自 I/O，不是演算法。

## 三、v4 改進內容

1. deterministic_io_optimize()：確定性文字替換
   - 加 ios_base::sync_with_stdio(false) + cin.tie(NULL)
   - endl → "\n"
   - 不需要 LLM，0 風險

2. Speedup guard：如果 CC 輸出比 naive 慢 (< 0.95x)
   - 回退到 naive + 確定性 IO
   - 確保「至少不會更慢」

3. Pipeline 變成：
   R1-R5 (正確性) → R6 (確定性IO) → R7 (LLM Pass2) → Speedup guard

## 四、Benchmark noise 問題

PIE 很多題 runtime < 10ms，benchmark 完全被 OS scheduling noise 支配。
例如 #1 p03161: src=3.6ms, opt=6.5ms → 顯示 0.55x，但這個差異
在 3ms 級別完全是噪音。

結論：PIE speedup 數字在短 runtime 題目上不可靠。
論文中需要：(a) 過濾 runtime > 50ms 的題目做 speedup 分析，
或 (b) 用 PIE 自帶的 aggregated runtime 數字。

## 五、Gemini 對論文方向的建議

Gemini 認為 73% I/O 發現本身就是貢獻。建議論文方向：

"Disentangling I/O Overheads from Algorithmic Optimization
 in Automated Code Refinement"

核心敘事：
1. 發現 PIE dataset 73% 的 speedup 來自 I/O（不是演算法）
2. 用確定性工具解決 I/O 層（不需 LLM）
3. 讓 LLM 專注剩下 27% 的演算法優化
4. 結果：7B 本地模型 94% pass rate (vs GPT-4 69%)

## 六、剩餘問題

1. 唯一失敗：p00431（圖論 DFS backtracking）— 7B 的認知天花板
2. Speedup guard 不夠嚴格：0.55x 的 case 沒被攔住（benchmark noise 導致）
3. 需要更大樣本量（目前 18-20 題，需 100+ 題）
4. 需要在 runtime > 50ms 的題目上做 speedup 分析
