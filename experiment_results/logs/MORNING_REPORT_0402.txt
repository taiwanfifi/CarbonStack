## 一、A/B Test 結果總覽

| 版本 | 說明 | Pass Rate | Median Speedup | Mean Speedup | Max |
|------|------|-----------|----------------|--------------|-----|
| v1 保守 (run1) | 只保持正確性 | 14/18 (78%) | 0.99x | 1.00x | 1.29x |
| v1 保守 (run2) | 重複驗證 | 14/18 (78%) | 1.03x | 1.03x | 1.29x |
| v2 激進 | 要求改演算法 | 5/9 (56%) | 0.86x | 0.92x | 1.04x |

## 二、關鍵發現

### 發現 1：兩段式策略 pass rate 最高 (83%)
v3 比 v1 多通過 1 題（p02946 或 p00729），pass rate 78%→83%。
原因：Pass 1 保守策略拿到正確 code，Pass 2 在不破壞的前提下嘗試優化。

### 發現 2：激進改演算法會大幅降低 pass rate
v2 直接要求「用更高效演算法重寫」→ 56% pass rate，4 題完全無輸出。
原因：7B 模型能力不足以一次完成「理解題目 + 改演算法 + 保持正確」。

### 發現 3：Speedup 在 10 題和 20 題間有差異
10 題 v3: 1.17x median vs 20 題 v3: 0.94x median。
原因：10 題的 seed 42 抽樣剛好是比較容易優化的題目。樣本效應。

### 發現 4：Pass 2 的 speedup 提升有限
Pass 2 在多數情況下沒有找到更快的版本，自動回退到 Pass 1。
原因：CP code 本身已經比較緊湊，7B 模型能找到的優化空間有限。

## 三、Bug 修復歷程（0% → 83%）

| 階段 | Bug | 修法 | 效果 |
|------|-----|------|------|
| 1 | bits/stdc++.h regex | 正則表達式匹配 | compile 60%→94% |
| 2 | Sanitizer 誤判 | 只查 ASAN/UBSAN keywords | pass ~30%→~50% |
| 3 | 單一測試案例驗證 | R4.6 multi-test validation | 結構性修復 |
| 4 | Timing 輸出干擾比對 | validate_output_match | 正確答案不再被誤判 |
| 5 | **Prompt 叫模型刪 stdin** | _detect_reads_stdin() | **0%→78%（根因）** |
| 6 | 兩段式策略 | Pass1正確 + Pass2優化 | 78%→83%, speedup↑ |

## 五、Gemini 討論摘要（4 輪）

### Round 1：確認 3 個 bug fix 正確
Gemini 100% 認同，預測修完後 15-30% pass rate。

### Round 2：深入討論 Verifier / Prompt / Error Feedback
- Verifier：加 property-based testing 和 complexity check
- Prompt：強制模型先分析 complexity 再寫 code
- Error feedback：給 Input+Expected+Actual，但不給 how to fix

### Round 3：A/B test 策略
- 建議先跑 50 題 baseline，再改 prompt，再跑大量
- 統計量至少 N=100

### Round 4：兩段式+AST 策略
- 完全認同 William 的 AST targeted optimization 想法
- 建議用 loop 層級而非 function 層級（CP code 大多在 main 裡）
- 風險：確保 Pass 2 不改 function signature/side effect
