## 一、專案背景

CarbonCode 是一個用本地 7B LLM（qwen2.5-coder:7b）自動優化 C++ 程式碼的
系統。核心目標是在完全 air-gapped 環境下（半導體/電信產業）實現程式碼效能優化，
不依賴任何雲端 API。

目標會議：NeurIPS 2026
目標場景：air-gapped deployment for semiconductor/telecom
使用模型：qwen2.5-coder:7b（本地 Ollama 部署）
評估基準：PIE dataset（ICLR 2024），978 道 C++ competitive programming 題目

為什麼選 7B？
- 半導體/電信產業有嚴格的 data isolation 要求，不能用雲端 LLM
- 7B 是單張消費級 GPU（RTX 3090 / RTX 4090）能跑的最大規模
- 如果 7B 能做到有意義的優化，代表真正能部署到產線上

為什麼選 PIE？
- ICLR 2024 accepted paper，社群認可的 benchmark
- 978 道題有完整的 test cases、原始碼、目標碼
- 可以量化 pass rate 和 speedup
- GPT-4 的結果是 69% pass@4，是我們要比較的 baseline


## 二、Pipeline 演進過程

=== 階段一：初版 CC Pipeline ===

五步驟架構：
  R1 — Analysis: 分析原始碼，找出效能瓶頸
  R2 — Design:   設計優化方案
  R3 — Generate: 生成優化後的程式碼
  R4 — Validate: 編譯並執行驗證
  R5 — Repair:   如果驗證失敗，自動修復

問題：7B 模型無法從高層描述推導正確演算法。
R1/R2 階段的「分析」和「設計」對 7B 來說太抽象。模型會產生看起來很有道理
但實際上有 logic error 的程式碼。特別是：
- 無法正確理解 loop invariant
- 無法推導 edge case
- 常常「理解」了問題但「實作」錯誤

=== 階段二：Decompose Pipeline ===

針對 7B 的能力限制，設計了 decompose pipeline：
- 將大問題拆解成小步驟
- 每個步驟都是 7B 能力範圍內的
- 在 CarbonBench v2 上有顯著效果（見第九節）

=== 階段三：PIE 整合 ===

從 0% compile rate 一路修到 94% pass rate 的完整 bug 修復歷程：

Bug 1: bits/stdc++.h regex
  症狀：compile rate 只有 60%
  原因：PIE 的 C++ 碼用了 #include <bits/stdc++.h>，這是 GCC 專用 header，
        在 macOS clang 上不存在
  修復：regex 替換為實際需要的 standard headers
  結果：compile rate 60% → 94%

Bug 2: Sanitizer false rejection
  症狀：很多程式碼編譯成功但被判定為 runtime error
  原因：開了 AddressSanitizer / UBSan，某些合法但 tricky 的 C++ 寫法被誤判
  修復：調整 sanitizer 配置，只在 debug mode 開啟
  結果：減少 false negative

Bug 3: Single-test validation gap
  症狀：通過第一個 test case 但在後續 test case 失敗
  原因：R4 階段只用了一個 test case 驗證
  修復：R4.6 — multi-test validation，用所有可用的 test cases
  結果：pass rate 更 robust

Bug 4: Timing output false mismatch
  症狀：output 比較失敗，但人眼看是一樣的
  原因：程式碼裡加了 timing output（如 "Time: 0.003s"），被當成程式輸出
  修復：strip timing output before comparison
  結果：消除 false mismatch

Bug 5（ROOT CAUSE — 最關鍵的 bug）:
  症狀：大量程式碼 compile 成功、但輸出完全錯誤
  原因：Prompt 告訴 model "use mt19937 seed 42 for deterministic testing"，
        模型理解為「不需要讀 stdin，自己生成測試資料」，於是刪除了所有
        cin/scanf，改用 mt19937 產生假資料
  修復：實作 _detect_reads_stdin() 函式，偵測原始碼是否讀取 stdin。
        如果是，使用 stdin-preserving prompt mode，明確告訴 model
        「保留所有 I/O 操作」
  結果：pass rate 大幅提升到 94%
  教訓：Prompt 必須匹配 code 的 I/O pattern，否則 7B model 會做出
        catastrophic 的改動


## 三、A/B Test 實驗結果

在 PIE dataset 的子集上跑了四個版本的對比實驗：

┌────────┬──────────────────────────────────┬──────────┬──────────┐
│ 版本   │ 策略                             │ Pass Rate│ Speedup  │
├────────┼──────────────────────────────────┼──────────┼──────────┤
│ v1     │ 保守 prompt（只改 I/O）          │ 78%      │ 1.0x     │
│ v2     │ 激進 prompt（要求改演算法）      │ 56%      │ 0.92x   │
│ v3     │ 兩段式（Pass1 正確 + Pass2 優化）│ 83%      │ 0.91x   │
│ v4     │ 確定性 I/O + speedup guard      │ 94%      │ 0.97x   │
└────────┴──────────────────────────────────┴──────────┴──────────┘

關鍵發現（KEY FINDING）：
  7B model 無法同時做 correctness 和 optimization。

v2 的失敗證明了這一點：當 prompt 要求「改演算法提升效能」時，model 的
pass rate 從 78% 掉到 56%，而且 speedup 反而 < 1.0（變慢了）。

v3 嘗試兩段式：第一段只保證正確性，第二段再優化。Pass rate 提升到 83%，
但 speedup 依然 < 1.0，代表第二段的優化其實在破壞正確性。

v4 是目前最佳版本：
- 確定性 I/O 優化（不靠 LLM，用 regex/AST 做 I/O 加速）
- Speedup guard（如果優化後比原始慢，回退到原始碼）
- 94% pass rate，0.97x speedup

v4 的 speedup 接近 1.0 但略低於 1.0，說明：
1. 確定性 I/O 優化是安全的（不會弄壞程式碼）
2. 但真正有意義的 speedup 還沒達到
3. 0.97x 代表有少量 case 是「改了但變慢一點」


## 四、PIE Dataset 深度分析（關鍵發現）

對 PIE 975 道題的 target code 做了全量分析，分類 source 和 target 之間的差異：

┌──────────────────────────┬──────────┬──────────┬────────────────┐
│ 改動類型                 │ 題數     │ 佔比     │ Median Speedup │
├──────────────────────────┼──────────┼──────────┼────────────────┤
│ 只改 I/O                 │ 194      │ 16%      │ 5.29x          │
│ I/O + 演算法都改         │ 264      │ 34%      │ 5.22x          │
│ 只改演算法               │ 326      │ 34%      │ 1.80x          │
│ 都沒大改                 │ 191      │ 16%      │ 1.86x          │
└──────────────────────────┴──────────┴──────────┴────────────────┘

深度觀察：

1. I/O 改動貢獻了 41% 的總時間節省
   - 這是一個 dataset bias：competitive programming 的 I/O 通常很慢
     （用 cin/cout without sync），而加上 ios_base::sync_with_stdio(false)
     和 '\n' instead of endl 就能大幅加速
   - 在真實 production code 中，I/O 不太可能是主要瓶頸

2. 純演算法改動中，真正複雜度降級（>10x speedup）只有 2%
   - 大部分「演算法改動」其實是 micro-optimization（const ref, reserve, etc.）
   - 真正的 O(n²) → O(n log n) 極少

3. PIE 論文宣稱 GPT-4 達到 3.56x speedup，但我們的分析顯示 I/O 加速佔了
   很大比重。如果剝離 I/O 改動，pure algorithmic speedup 會低很多。

4. 之前的分析錯誤修正：
   - 錯誤宣稱：73% 是 I/O 改動
   - 修正為：16% 是純 I/O，50% 涉及 I/O（包含 I/O + 演算法混合）
   - 這個修正很重要，影響論文的 narrative

5. 「都沒大改」的 191 題 median speedup 1.86x
   - 這些可能是 compiler optimization 或 micro-optimization
   - 也可能是 noise（PIE 的 timing 有 variance）


## 五、我們實際做到的 vs 還沒做到的

=== 已做到 ===

1. 94% pass rate (pass@1)
   - vs GPT-4 的 69% (pass@4)
   - 這是一個顯著的結果：本地 7B 的 pass@1 > 雲端 GPT-4 的 pass@4
   - 但需要注意：我們的 pass rate 是「不弄壞程式碼」，不是「加速程式碼」

2. 確定性 I/O 優化
   - 不需要 LLM 就能做的 I/O 加速
   - ios_base::sync_with_stdio(false)
   - cin.tie(nullptr)
   - '\n' instead of endl
   - printf/scanf instead of cin/cout（在某些 case）

3. 兩段式安全架構
   - Pass 1: 確保 correctness
   - Pass 2: 嘗試 optimization
   - 如果 Pass 2 失敗，回退到 Pass 1 的結果

4. Speedup guard
   - 如果優化後的版本比原始版本慢，自動回退
   - 保證「至少不會變慢」

5. Multi-test validation (R4.6)
   - 使用所有可用的 test cases 驗證
   - 大幅減少 false positive

6. _detect_reads_stdin()
   - 偵測原始碼是否讀取 stdin
   - 根據結果選擇不同的 prompt mode
   - 解決了 Bug 5（最關鍵的 root cause bug）

=== 還沒做到 ===

1. 真正的 speedup > 1.5x
   - 目前的 median speedup 接近 1.0x
   - I/O 優化能貢獻一些，但不夠

2. 演算法複雜度降級
   - 7B model 無法做 O(n²) → O(n log n) 的轉換
   - 這需要理解 loop structure 和 data dependency

3. 讓 7B 理解 loop 的 Big-O
   - 7B 可以「說出」Big-O 是什麼
   - 但無法「基於 Big-O 分析」做出正確的程式碼改動


## 六、Gemini 討論摘要（多輪諮詢）

與 Gemini 進行了多輪技術討論，摘要如下：

Round 1: Bug Fix 驗證
  - 確認 Bug 1-5 的修復邏輯正確
  - Gemini 同意 _detect_reads_stdin() 是關鍵修復
  - 建議增加 regression test

Round 2: Verifier Quality & Prompt Engineering
  - 討論 R4 verifier 的 false positive/negative rate
  - Prompt engineering 的系統化方法
  - Error feedback loop: 把錯誤訊息餵回 model 讓它修正

Round 3: Two-Pass Strategy & Sample Size
  - 兩段式策略的理論基礎
  - 論文需要的 sample size：978 題全跑才有說服力
  - Statistical significance testing 方法

Round 4: AST Targeted Optimization & Contract-Based Assembly
  - William 提出的 AST targeted optimization 概念
  - 用 AST 識別 hot loop，只對特定 code region 做 LLM 優化
  - Contract-based assembly: 定義 pre/post condition，讓 model 在 contract
    框架內優化

Round 5: Cargo Cult Effect（重要概念）
  - 7B model 學到的是 coding pattern 的「美學」而非「機制」
  - 例如：model 會把 vector 改成 array 因為「看起來更快」
  - 但不理解為什麼 array 更快（cache locality, no dynamic allocation）
  - 這導致 model 的優化有時候反而變慢
  - Cargo cult effect 是解釋 7B limitation 的好框架

Round 6: I/O Finding 改變論文 Narrative
  - 原本的 narrative: "7B can optimize C++ code"
  - 修正後的 narrative: "PIE benchmark has significant I/O bias, and
    even simple deterministic transforms capture much of the speedup"
  - 這其實是更強的論文貢獻：揭示 benchmark 的 composition


## 七、未來方向（William + Claude + Gemini 共識）

以下是三方討論後達成共識的未來方向，按優先級排列：

1. Decompose Pipeline 用於演算法優化
   - 將「優化這個 O(n²) 演算法」拆解成：
     a. 識別 bottleneck loop
     b. 分析 data dependency
     c. 選擇替代演算法
     d. 生成新 code
     e. 驗證正確性
   - 每步都是 7B 能做的

2. RAG 餵演算法模板
   - 建立 algorithmic pattern 的 knowledge base
   - 例如：sort → merge sort, search → binary search
   - RAG 檢索最相關的 pattern 餵給 model
   - 減少 model 需要「發明」演算法的負擔

3. AST Targeted Optimization（William 的想法）
   - 用 static analysis 找出 hot path
   - 只把 hot path 的 code snippet 餵給 LLM
   - 減少 context 大小，提升 model 的 focus
   - 可以用 perf/gprof 的 profiling 結果指導

4. Pre-LLM Triage: Static Analysis 判斷是否有優化空間
   - 在呼叫 LLM 之前，先用 static analysis 判斷
   - 如果 code 已經很 optimal（沒有明顯的 O(n²) 等），就 skip
   - 節省 LLM inference 時間

5. Self-Curriculum: 先解簡單版再解難版
   - 讓 model 先在簡單的 optimization 上練習
   - 逐步增加難度
   - 需要一個 difficulty scoring function

6. Hierarchical Self-Decomposition
   - 讓 model 自己決定如何 decompose
   - 類似 chain-of-thought 但更 structured
   - 每一層都有 verification

7. Reward = PassRate × Speedup
   - 同時獎勵正確性和效能
   - 可以用於 RLHF 或 best-of-N sampling
   - 當前的問題是兩者互斥（見第三節的 A/B test）


## 八、論文可能的亮點

按貢獻度排序，以下是可能的論文亮點：

1. PIE Dataset Composition Analysis（最強貢獻）
   - 首次對 PIE 975 題做全量分類分析
   - 揭示 I/O vs algorithmic optimization 的比例
   - 挑戰 PIE paper 的 implied narrative（GPT-4 的 3.56x 很大程度靠 I/O）
   - 這是 community contribution，即使我們的 model 效果不好，這個分析本身有價值

2. Two-Pass Architecture for Small Models
   - 證明 7B model 無法同時做 correctness + optimization
   - Two-pass 是解決方案
   - 有 A/B test 數據支撐

3. Deterministic + LLM Hybrid Optimization
   - 確定性 I/O 優化 + LLM 演算法優化
   - 混合架構比純 LLM 更 robust
   - 可以推廣到其他 code optimization 場景

4. 7B pass@1 94% vs GPT-4 pass@4 69%
   - 驚人的結果，但需要 careful framing
   - 我們的 94% 是「保持正確性」的 pass rate
   - GPT-4 的 69% 是「嘗試優化後仍然正確」的 pass rate
   - 目標不同，但仍然是有意義的比較

5. Speedup Guard Mechanism
   - 保證優化後不比原始慢
   - 在 production 場景很重要
   - 簡單但有效

6. _detect_reads_stdin: Prompt Must Match Code's I/O Pattern
   - 通用教訓：prompt 必須根據 code 的特性動態調整
   - 不是 one-size-fits-all
   - Bug 5 的案例分析本身就是一個好的 lesson learned


## 九、CarbonBench v2 結果

CarbonBench v2 是我們自製的 10 道任務，使用 mt19937(42) 生成自含測試資料。

Decompose Pipeline 結果：
  - Prefix Sum: 1080x speedup（O(n²) → O(n) with prefix sum）
  - Top-K:      2x speedup
  - 其他任務：不同程度的改善

觀察：
  - Decompose pipeline 在「有明確演算法改進空間」的任務上效果極好
  - 但對「已經接近最優」的簡單任務會過度設計（over-engineering）
  - 過度設計的案例：model 會把簡單的 for loop 改成複雜的 template
    metaprogramming，反而變慢

CarbonBench v2 的限制：
  - 只有 10 道題，sample size 太小
  - mt19937(42) 自含資料代表不需要 stdin，跟 PIE 不同
  - 需要在 PIE 978 題上驗證 decompose pipeline 的效果

## 十一、IO 正規化實驗結果（2026-04-02）

### 關鍵發現：PIE test case 太小，本地無法重現 speedup

所有 PIE 題目在我們機器上跑 3-5ms，src/tgt/fastest 完全沒有可測量差異。
原因：PIE test case 極小（N=2~5, 字串 7~40 字元, 圖 7 節點）。
PIE paper 的 5.3x speedup 是在他們的伺服器上用不同環境量的。

### 數據
- IO 優化效果: median 1.06x（noise 範圍內）
- IO 正規化後 algo gap: median 1.02x（幾乎為 0）
- tgt 比 src+IO 快 >1.5x 的: 0/18 題
- 結論: 在 3-5ms runtime 下，所有 speedup 差異都是 OS noise

### 影響
1. 我們不能用本地 benchmark 來 claim speedup — 必須用 PIE 自帶的 runtime 或放大輸入
2. Gemini 建議: 放大輸入到 100ms+ range，或用 looping/batching
3. 論文中需要說明 benchmark 環境差異

### PIE test case 大小
- p03161: N=2~5 (DP 問題, 輸入 11-19 bytes)
- p02676: 字串 7~40 字元 (18-42 bytes)
- p00431: 圖 7 節點 (88 bytes)

## 十二、CC + Decompose 整合結果（2026-04-02）

### PIE "both" method (5 tasks):
- Pass rate: 4/4 (100%)
- PIE test case 太小量不出 speedup 差異

### CarbonBench v2 Decompose（真正的 speedup 證據）:
| Task | Naive | Decompose | Speedup |
|------|-------|-----------|---------|
| Top-K | 0.34s | 0.098s | 3.4x |
| Prefix Sum | 10.46s | 0.0084s | 1244x |
| Two Pointer | 0.038s | 0.0077s | 4.9x |

Prefix Sum 1244x = O(N×Q) → O(N+Q) 的真正複雜度降級。
這是用 CarbonBench（有大輸入）才量得出來的。

### 論文敘事確定：
- PIE benchmark: 94% pass@1 correctness（證明 7B 能安全改 code）
- CarbonBench v2: 3.4x-1244x speedup（證明 Decompose 能做演算法優化）

## 十五、全部 22 題失敗案例逐一分析（2026-04-02）

### 結果
- 21 題跑完（1 題無 PIE entry）
- 救回 10 題（之前失敗現在成功）→ pass rate 可達 78+10=88/100 (88%)
- 仍然失敗 11 題，全部是 code_generated_but_tc_fail

### 關鍵發現：NO_OUTPUT 問題大部分是 LLM 隨機性
- 之前 GPU 上 22 題全部 no_output
- 重跑一次，10 題居然通過了（LLM 輸出有隨機性）
- 剩下 11 題才是真正的「解不了」

### 仍然失敗的 11 題特徵：
全部是 code_generated_but_tc_fail（模型生成了 code 但 test case 失敗）

| 特徵 | 題數 | 佔比 |
|------|------|------|
| 圖論 + 遞迴 + multi_io | 2 | p00431 (DFS backtracking) |
| DP | 3 | p03161 variants |
| 複雜巨集 (>7 #define) | 4 | p03161, p02898, p03313 |
| 短但邏輯難 | 2 | p02946 (14行), p02792 |

### 核心結論
1. **50% 的失敗是 LLM 隨機性**（重跑就通過）→ Best-of-N 可以解
2. **真正的瓶頸是 TC fail**（code 生成了但邏輯錯）→ 7B 的推理極限
3. **圖論 DFS + backtracking** 是 7B 的認知天花板
4. **複雜巨集** 干擾模型理解 → 巨集展開可能有幫助
5. **短 code 也可能失敗** → 不只是長度問題，也是邏輯複雜度

### 改善建議
1. Best-of-3: 跑 3 次取最好 → 消除隨機性失敗 → 預估 88%+
2. 巨集展開: #define 先展開再送給 LLM → 減少巨集相關失敗
3. 圖論專用 Decompose: THINK→PSEUDOCODE 更適合 DFS 類型

## 十六、巨集展開 A/B test 結果（2026-04-02）

### 結果：巨集展開反而降低 pass rate
- v5 (無巨集展開): 78/100 (78%) ← 最好
- v6 (全部巨集展開): 70/100 (70%) ← 降了 8%

### 詳細對比
- 巨集展開救了 12 題（之前失敗現在成功）
- 但搞壞了 19 題（之前成功現在失敗）
- 淨效果: -7 題

### 原因分析
巨集展開會改變 code 的結構和可讀性。某些展開（如 `#define ll long long` → `long long`）
是安全的，但其他展開（如 `#define MEM(x,y) memset(x,y,sizeof(x))`）可能破壞語法。
無差別展開不可行。

### 修正
改為只在 #define > 7 個的題目才展開（這些是失敗相關的高巨集題目）。
保留 78% 作為最佳 pass@1 結果。

### 教訓
- 不是所有「聽起來合理」的改動都有效 — A/B test 是必要的
- 巨集展開需要更精細的控制（只展開 type alias，不展開 function-like macros）
- 這個 negative result 本身也是論文材料（ablation study）

=== Finding #46: Carbon-Verify — Testbench Compiles but Doesn't Catch Bugs ===
Date: 2026-04-05

Improved prompt → both counter + adder testbenches now compile and run.
Previous: 1/3 compiled. Now: 3/3 compiled ✅

BUT: neither buggy module's bug was caught.
- counter: LLM expected reset=1 (matching the buggy code, not the spec)
- adder: LLM only displayed results, no expected value assertion

ROOT CAUSE: LLM generates testbench from CODE, not from SPEC.
If the code has a bug, the testbench will match the buggy behavior.

This is a fundamental limitation:
"LLM-generated testbenches verify that code does what it does,
not that code does what it SHOULD do."

To catch bugs, the testbench must be generated from:
1. A separate specification (not the code)
2. Explicit expected values in the prompt
3. Property-based assertions (e.g., "reset should always set count to 0")

This finding shapes Paper 5 (Carbon-Verify):
"Spec-driven testbench generation vs code-driven: why LLMs can't find bugs
they helped create."

=== Finding #47: Context Length vs PFS (Paper 2 supplement) ===
Date: 2026-04-05

Experiment: Vary number of few-shot examples (0, 1, 2, 3, 5) on DCT kernel.
Model: Gemma4, single kernel (DCT).

Results:
  0 examples → PFS = 1.000 (prompt 548 chars)
  1 example  → PFS = 1.000 (prompt 769 chars)
  2 examples → PFS = 1.000 (prompt 903 chars)  
  3 examples → PFS = 0.667 (prompt 1023 chars) ← DISTRACTION!
  5 examples → PFS = 1.000 (prompt 1277 chars) ← recovered (stochastic)

Observation: Distraction occurs non-monotonically. 3 examples caused
a significant drop (1.000→0.667) but 5 examples recovered.
This suggests distraction is PROBABILISTIC, not deterministic —
more examples increase the RISK of distraction but don't guarantee it.

For Paper 2: "Context Distraction is a stochastic phenomenon that
increases in probability with context length, manifesting as sporadic
attention failures rather than gradual degradation."

Need: more kernels × more seeds to establish statistical curve.
